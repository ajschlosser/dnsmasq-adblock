# dnsmasq-adblock

Lightweight DNS-level adblocking using `dnsmasq` in Docker.

This project runs `dnsmasq` in a container, downloads hosts-format blocklists at startup, converts them into `dnsmasq` rules, and serves DNS on a chosen host IP address.

## What this does

Normal DNS flow:

```text
device → this dnsmasq container → upstream DNS resolver → internet
```

Blocked domain flow:

```text
device → this dnsmasq container → 0.0.0.0 / ::
```

## What this does not do

This does not block every ad. DNS-level blocking only blocks whole domains.

It will not reliably block:

- YouTube ads
- first-party ads served from the same domain as the site
- browser traffic running through a VPN that uses its own DNS
- browser DNS-over-HTTPS if enabled

For browser-level blocking, use this together with uBlock Origin.

## Files

### `.env`

Controls where Docker exposes DNS.

```env
DNS_BIND_IP=10.0.0.98
```

Use the stable IP address of the host running this container.

Avoid `0.0.0.0` unless you intentionally want this DNS server exposed on every host interface.

### `Dockerfile`

Builds the container image. It installs:

- `dnsmasq`
- `bash`
- `curl`
- `ca-certificates`
- `tini`

You usually only need to touch this if you are changing the base image or adding packages.

### `docker-compose.yml`

Defines the container, port bindings, mounted files, and restart behavior.

The important part is:

```yaml
ports:
  - "${DNS_BIND_IP}:53:53/udp"
  - "${DNS_BIND_IP}:53:53/tcp"
```

This exposes container DNS on the host IP configured in `.env`.

### `config/dnsmasq.conf`

Main `dnsmasq` configuration.

Most day-to-day changes should not go here. Prefer:

```text
config/upstream.conf
config/local.conf
config/blocklist-urls.txt
```

### `config/upstream.conf`

Upstream DNS resolvers used after local block rules are applied.

Default:

```ini
server=1.1.1.1
server=1.0.0.1
server=9.9.9.9
server=149.112.112.112
```

`1.1.1.1` and `1.0.0.1` are Cloudflare public DNS.

`9.9.9.9` and `149.112.112.112` are Quad9 public DNS.

Edit this file if you want different upstream resolvers.

### `config/blocklist-urls.txt`

One blocklist URL per line.

Blank lines and comments are ignored.

Example:

```text
# Ads and trackers
https://raw.githubusercontent.com/StevenBlack/hosts/master/hosts
https://adaway.org/hosts.txt
https://big.oisd.nl/hosts
```

Edit this file to add, remove, or change blocklist sources.

### `config/local.conf`

Manual `dnsmasq` rules.

Use this for one-off blocks, local overrides, or internal names.

Examples:

```ini
address=/bad-domain.example/0.0.0.0
address=/bad-domain.example/::
address=/router.home/10.0.0.1
server=/example.com/1.1.1.1
```

### `generated/blocklist.conf`

Generated blocklist rules.

Do not edit this file by hand. It is overwritten every time the container starts.

It is mounted on the host so you can inspect/debug the generated `dnsmasq` rules without entering the container.

### `scripts/entrypoint.sh`

Container startup script.

It:

1. Reads `config/blocklist-urls.txt`
2. Downloads blocklists
3. Converts hosts-format entries into `dnsmasq` rules
4. Writes `generated/blocklist.conf`
5. Starts `dnsmasq`

## Setup

Create the project directories:

```bash
mkdir -p dnsmasq-adblock/config dnsmasq-adblock/scripts dnsmasq-adblock/generated
cd dnsmasq-adblock
```

Create the required files:

```bash
touch Dockerfile docker-compose.yml README.md .env .gitignore
touch config/dnsmasq.conf config/upstream.conf config/local.conf config/blocklist-urls.txt
touch generated/blocklist.conf
touch scripts/entrypoint.sh
chmod +x scripts/entrypoint.sh
```

Set the host IP in `.env`:

```env
DNS_BIND_IP=10.0.0.98
```

Verify the host actually has that IP:

```bash
ip -4 addr
```

Make sure port 53 is not already occupied on that IP:

```bash
sudo ss -tulpn | grep ':53'
```

It is usually fine if `systemd-resolved` is bound only to:

```text
127.0.0.53:53
127.0.0.54:53
```

But nothing else should already be bound to:

```text
10.0.0.98:53
```

Replace `10.0.0.98` with your actual `DNS_BIND_IP`.

## Test DNS directly

Install `dig` if needed.

Fedora:

```bash
sudo dnf install -y bind-utils
```

Debian/Ubuntu:

```bash
sudo apt update
sudo apt install -y dnsutils
```

Test a normal domain:

```bash
dig @10.0.0.98 google.com
```

Test a blocked domain:

```bash
dig A @10.0.0.98 doubleclick.net
dig AAAA @10.0.0.98 doubleclick.net
```

Expected blocked results:

```text
A    → 0.0.0.0
AAAA → ::
```

Replace `10.0.0.98` with your actual `DNS_BIND_IP`.

## Allow DNS through the host firewall

After the container is running, the host firewall may still block DNS traffic from other devices.

### firewalld

Fedora commonly uses `firewalld`.

Find the active zone:

```bash
sudo firewall-cmd --get-active-zones
```

Example output:

```text
FedoraWorkstation
  interfaces: wlp6s0
```

Allow DNS on that zone:

```bash
sudo firewall-cmd --zone=FedoraWorkstation --add-service=dns --permanent
sudo firewall-cmd --reload
```

If your active zone is `home`, use:

```bash
sudo firewall-cmd --zone=home --add-service=dns --permanent
sudo firewall-cmd --reload
```

If your active zone is `public`, use:

```bash
sudo firewall-cmd --zone=public --add-service=dns --permanent
sudo firewall-cmd --reload
```

### ufw

Ubuntu commonly uses `ufw`.

Check whether `ufw` is active:

```bash
sudo ufw status
```

If `ufw` is active, allow DNS:

```bash
sudo ufw allow 53/udp
sudo ufw allow 53/tcp
```

### Test from another device

From another device on the same network:

```bash
nslookup google.com 10.0.0.98
nslookup doubleclick.net 10.0.0.98
```

Replace `10.0.0.98` with your actual `DNS_BIND_IP`.

## Make the host itself use the container

This is optional. It makes the machine running the container also use the container for DNS.

Find your active NetworkManager connection:

```bash
nmcli connection show --active
```

Set DNS for that connection:

```bash
sudo nmcli connection modify "YOUR_CONNECTION_NAME" ipv4.ignore-auto-dns yes
sudo nmcli connection modify "YOUR_CONNECTION_NAME" ipv4.dns "10.0.0.98"
sudo nmcli connection modify "YOUR_CONNECTION_NAME" ipv6.ignore-auto-dns yes
sudo nmcli connection modify "YOUR_CONNECTION_NAME" ipv6.dns ""
sudo nmcli connection down "YOUR_CONNECTION_NAME"
sudo nmcli connection up "YOUR_CONNECTION_NAME"
```

Flush resolver cache if `systemd-resolved` is running:

```bash
sudo resolvectl flush-caches
```

Verify:

```bash
resolvectl status
dig doubleclick.net
```

`dig doubleclick.net` should return `0.0.0.0` and/or `::`.

If NetworkManager is not managing DNS, check your distro’s network stack before editing files directly. Common places include:

```text
/etc/systemd/resolved.conf
/etc/netplan/*.yaml
/etc/network/interfaces
```

For most desktop Fedora, Ubuntu, and Debian installs, NetworkManager is the easiest and safest path.

## Router/client configuration

Once the DNS server works locally, point clients or your router DHCP DNS to:

```text
10.0.0.98
```

Replace with your actual `DNS_BIND_IP`.

Recommended router DHCP DNS:

```text
Primary DNS:   10.0.0.98
Secondary DNS: blank, or 10.0.0.98
```

Do not configure `1.1.1.1`, `8.8.8.8`, Xfinity DNS, or another public resolver as secondary DNS unless bypassing adblocking is acceptable. Many clients use secondary DNS opportunistically.

## Troubleshooting

### Container logs show permission denied for `entrypoint.sh`

Make sure the script is executable:

```bash
chmod +x scripts/entrypoint.sh
docker compose up -d --force-recreate
```

On SELinux systems, use `:Z` on bind mounts in `docker-compose.yml`.

### Container cannot write `blocklist.conf`

Use the `generated/` directory mount:

```yaml
- ./generated:/generated:Z
```

Make sure `config/dnsmasq.conf` references:

```ini
conf-file=/generated/blocklist.conf
```

Make sure `scripts/entrypoint.sh` writes to:

```bash
BLOCKLIST="/generated/blocklist.conf"
```

### `dig @<DNS_BIND_IP> google.com` times out

Check that Docker is publishing port 53:

```bash
docker ps --filter name=dnsmasq-adblock
sudo ss -tulpn | grep ':53'
```

You should see something like:

```text
10.0.0.98:53->53/tcp
10.0.0.98:53->53/udp
```

Check the container logs:

```bash
docker logs --tail=100 dnsmasq-adblock
```

Check whether `dnsmasq` works inside the container:

```bash
docker exec -it dnsmasq-adblock sh
nslookup google.com 127.0.0.1
```

If it fails inside the container, check `config/dnsmasq.conf`.

A working listener section should include:

```ini
interface=lo
interface=eth0
bind-interfaces
```

not:

```ini
listen-address=0.0.0.0
```

### DNS works with `dig @<DNS_BIND_IP>` but not normally

Your host or device may not be using this DNS server.

Check the active resolver:

```bash
resolvectl status
nmcli connection show --active
```

Set the active connection DNS to your `DNS_BIND_IP`.

### DNS works in terminal but Firefox still shows ads

Check:

- Firefox DNS-over-HTTPS
- Firefox VPN
- proxy settings
- browser cache

Firefox VPN and many other VPNs use their own DNS path, bypassing this container.

Also, DNS blocking is not browser content blocking. For modern websites, use uBlock Origin in addition to DNS-level blocking.

### `doubleclick.net` is blocked but ads still appear

That can be normal.

Some ads are first-party, served from the same domain as the website, or loaded through domains you cannot block without breaking the site.

DNS-level blocking is a baseline filter. Browser-level blocking is still useful.
